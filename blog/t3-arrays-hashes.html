    • Arrays and Hashes

    Whats the difference?

    The best way to summarize the coparison between these two types of objects is that one, hash, is a more specific version of an array. An array has one set of values, while a hash has two sets of values. An array is a group or elements, while a hash is type of collection of elements that contain keys and values.

    Think of a hash as two arrays that are related to each other. Sort of like a table of contents in a book:

    Table of Contents
    Prefix ............................ 3
    Chapter 1 ......................... 23
    Chapter 2 ......................... 45
    Chapter 3 ......................... 53
    Chapter 4 ......................... 78
    Chapter 5 ......................... 123
    Chapter 6 ......................... 168
    Chapter 7 ......................... 201
    Chapter 8 ......................... 222

    One array, the Key array, could be seen as the chapter titles of the book, so if the Table of Contents was a hash (lets call it 'toc'), the keys of 'toc' would be an array that looks like this :

    [ prefix, Chapter 1, Chapter 2, Chapter 3, Chapter 4, Chapter 5, Chapter 6, Chapter 7, Chapter 8 ]

    while the Values array could be the page numbers that each title corresponds to. The array for the values would look like this:

    [ 3, 23, 45, 53, 78, 123, 168, 201, 222 ]

    And these two arrays would be related though what index they each sit at in their respective arrays.

    So to close out this thought, a hash can be thought of as TWO PARALLEL ARRAYS, whos elements are invariably linked to one another by position in their respective array. An array can be thought of as a hash, in which the keys are automatically incrementing numbers starting at 0 for the first element (incrementing by one each position), and values which represent the elements in the array. You may call elements in an array by referencing its position, (ie Array[3] == 4th element), and you may call values in a hash by referencing its key (ie Hash[Key] == the values corresponding to the key)

    Declaring

    First thing is first, declaring each is done differently. An array, for example is initiated with a pair of square brackets ([]). If we wanted to initiate an array called 'mat' we would type something like this:

    mat=[]

    This would give us an array of length 0 called mat.

    A Hash on the other hand is declared with curly braces ({}). So the get a hash called 'mat' of length 0, would be done like this:

    mat={}

    Populating and selecting

    Selecting various elements in these objects is very important. In an array, you have to select a certain element by referncing its order (index) in the array.

    Array may be populated with any type of object, (even other arrays!). An array of strings looks like ['string1', 'string2'], one of numbers would look like [1,2,3]. Unlike other programming languages, arrays can be of mixed object types, although its not recommending, because a fiar amout of errors wil pop up if you try to transverse that array using a string method, or number method

    Array elements are assigned by thier index values, while elements in hashes are assigned and selected by their key values.

    There are plenty of ways of making a hash, this most simply way is to fill a hash with key/value elements, the key and value are seperated by the => symbol, and treat those as elements to be entered in the array.

    Objects in a hash look like this:

    K=>V

    K stands for what key this particular element is, and V is the value of that element. The value is assigned and selected by the key. The key replaces the index is terms of selecting and assigning.

    Uses

    Now that we have properly distinguished between these two types of objects, Arrays and Hashes, now we delve into why they exist. What can you do with a hash that you can't do with an array?

    collections

    First off, the Hash class, is a subclass of the Collection class. This means, among other things, it automatically removes duplicates for us. In an array, the .uniq method would need to be used to do that, but since Hash is a subclass of Collection, it automatically does that for us, which makes alot of sense because if you think about it, the way elements are selected in a hash does not leave room for uplicates. If we have two identical sets of keys and values and we call one key, which element is really being called? If two of them exist, are will be get two values? but the values are also the same, so maybe we'll get one value? Hashes protect themselves from these types of paradoxes by auto-eleminating all duplicates.

    enumerables

    Another way hashes make our lives easier is the ability to sort by either keys or values, via an enumerable. If I have a hash that describes the number of people that were born in certain various population levels (population levels being the keys, and that values being the number of births) and I wanted to find the population that the most births took place in, then I could sort this hash by the number of births, and grab the key of the last element. Or the other way, if I wanted to find how many births took place in the population with the least number of people, then I could sort by population, and find the value of the last element.

    describing objects: Matt=[] vs. Matt={}

    Objects in Ruby are basically sets of properties, and can be described in two ways: Arrays and Hashes. Lets take an object modeled from myself.

    If I were to describe myself in an Array, it would look something like this:

    Matt=['Matt', 'Lao', 'San Francisco', 'DBC student']

    Now I can use the object Matt as one object with a set of properties described in an array. My first name would be Matt[0], my ocupation would be Matt[3] and so on.

    A hash description of me would look like this:

    Matt={first=>'Matt', last=>'Lao', location=>'San Francisco', ocupation=>'DBC student'}

    Now I can call specifics about my self. Such as Matt[first] for my first name regardless of order:

    Matt={first=>'Matt', last=>'Lao', location=>'San Francisco', ocupation=>'DBC student'}

    -- is quivalent to --

    Matt={ ocupation=>'DBC student', last=>'Lao', location=>'San Francisco',first=>'Matt' }

    While this may not always be nessesary, it makes like ALOT simpler.

    Making one from the other

    Using arrays to make hashes

    There are a number of ways that hashes may be made of arrays (without using a loop). Here I will talk about two of them.

    The first way is to use an array of keys and values in a particular order, [key1,value1,key2,value2,key3,value3] (so we always need an even number of arguments). In this way we can define as hash called car, with a set of properties:

    mycar=['Make','Honda','Model','Civic','Year',2003]

    In order to make this as hash we must wrap it in a Hash and ruby will read it as a hash

    Hash['Make','Honda','Model','Civic','Year',2003]

    => {"Make"=>"Honda", "Model"=>"Civic", "Year"=>2003}

    This is useful if you have a method that only spits out an array, and wish to turn that into a hash

    Another way to do this is to marry two arrays together : Hash=[keys.zip(values)]

    car_properties=['Make','Model','Year']
    car_description=['Honda','Civic',2003]
    car_properties.zip(car_description)
    => [["Make", "Honda"], ["Model", "Civic"], ["Year", 2003]]

    This leades us into a thirdway of making the hash. Notice how instead of a regular looking hash, we have an array of arrays. Also called a multi-dimentional array. In order for us to make keys of the first elements in each array element, and values of the second elements, we need to wrap the whole thing in an extra pair of square braces, and a Hash:

    Hash[[["Make", "Honda"], ["Model", "Civic"], ["Year", 2003]]]

    => {"Make"=>"Honda", "Model"=>"Civic", "Year"=>2003}

    Using hashes to make arrays

    Making arrays from hashes is as simple as the .keys and .values methods. The .keys method will give you an array of keys, while the .values method gives you an array of........ wait for it..... VALUES!!! This is the shortest part of my blog for a reason, its the most straight foreward thing to think about. Remember, like I stated earlier, hashes can be thought of as sets of two parrallel arrays, whos elements are related to each other via index position.
